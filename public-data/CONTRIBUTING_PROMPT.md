# Contributing a public dataset to Chronicle

Public datasets are world/topic timelines (election terms, product release histories,
historical eras…) that everyone sees alongside their private data. Most files here are
generated by pasting the prompt below into an LLM, then reviewing the result.

## How to contribute

1. Copy the prompt template below, replace `{{TOPIC}}` (and optionally `{{NOTES}}`).
2. Paste it into any capable LLM and save the emitted JSON as
   `public-data/<topic-slug>.json` (kebab-case, e.g. `german-chancellors.json`).
3. Check the facts. The LLM will get dates wrong; you are the editor.
4. Open a PR. CI validates the file against `public-data/schema.json`.

Ids only need to be unique **within your file** — the app automatically prefixes every id
(and every reference to one) with `pub:<filename>:` when loading, so don't worry about
colliding with other files.

## Prompt template

---

Produce a single JSON file describing a timeline dataset about: **{{TOPIC}}**.

Additional notes: {{NOTES}}

The JSON must conform exactly to this structure (no extra fields anywhere):

```json
{
  "schemaVersion": 1,
  "groups": [{ "id": "g-1", "label": "<topic name>", "collapsed": false }],
  "categories": [
    {
      "id": "cat-1",
      "label": "<category name>",
      "color": "<any CSS color, pick something fitting the topic>",
      "icon": "<one fitting emoji>",
      "concurrency": "exclusive or concurrent",
      "defaultVisibility": "shareable"
    }
  ],
  "rows": [
    { "id": "row-1", "groupId": "g-1", "categoryId": "cat-1", "label": "<row label>" }
  ],
  "entities": [
    { "id": "ent-1", "kind": "person|place|organization|object|other", "label": "<name>" }
  ],
  "entries": [
    {
      "id": "e-1",
      "rowId": "row-1",
      "title": "<short title>",
      "description": "<one or two factual sentences>",
      "start": { "ms": 0, "precision": "day" },
      "end": { "ms": 0, "precision": "day" },
      "linkedEntityIds": ["ent-1"]
    }
  ]
}
```

Rules:

- Every `ms` value is UTC milliseconds since the Unix epoch, i.e. the result of
  `Date.UTC(year, monthIndex, day)`. Compute them carefully.
- `precision` expresses how well the date is known: `exact` (to the minute), `day`,
  `month`, `year`, or `circa` (rough). Use the coarsest precision that is honest —
  for a "sometime in 1994" event use `year` with `ms` at mid-1994.
- Omit `end` entirely for things that are still ongoing.
- Consecutive terms/holders/versions on the same row: use an `exclusive` category and
  make each entry's `end` equal the next entry's `start`.
- Ids: short and unique within this file only (`g-1`, `cat-1`, `row-1`, `ent-1`, `e-1`, …).
- Link entries to the people/places/organizations they involve via `entities` +
  `linkedEntityIds`.
- Output ONLY the JSON, no commentary, no markdown fences.

---

## Checklist before opening the PR

- [ ] File name is `<topic-slug>.json` and ids are unique within the file.
- [ ] Dates spot-checked against a reliable source.
- [ ] Validates against `schema.json` (CI runs this; locally: `npx vitest run src/publicData`).
